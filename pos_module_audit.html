EODEnd-of-Day module
!
cash_in / cash_out columns do not exist
eodPdf.js references r.cash_in and r.cash_out in the shift reconciliation formula, but neither column is in the eod_reports table, the Rust model, or the migration. The expected-cash calculation in the PDF will always be wrong — it silently defaults to zero, producing a false reconciliation.
Critical
!
Multi-shift day only captures the last shift
generate_eod_report fetches opening_float / closing_cash with ORDER BY closed_at DESC LIMIT 1. If a store runs morning + evening shifts, only the last shift is reflected. Cash reconciliation for multi-shift days will be inaccurate. Need to aggregate across all shifts for the date.
Critical
!
Generate action uses analytics.read permission
Generating a report is a write/upsert operation (INSERT ... ON CONFLICT DO UPDATE). Using analytics.read means a read-only analyst role can overwrite an existing EOD report. Should be reports.generate or at minimum analytics.write.

No guard against future-date generation
The backend accepts any date string including future dates. A manager could accidentally generate a ₦0 EOD for tomorrow and lock it. Add a validation: if date_str > today { return Err(...) }.

Locking is irreversible. The current UI fires lock.mutateAsync(id) on a single click with no confirmation step. At minimum a short confirmation popover should be shown. Compare: the Void Transaction flow has a 2-step PIN-confirm dialog for the same reason.

The 4-tile KPI row shows Gross Sales, Net Sales, Transactions, Top Seller. A professional POS EOD should show at minimum: gross profit margin % (gross_profit / gross_sales) and items sold (already on the model). Top Seller belongs in the Products tab, not a KPI card.

Priority fixes
1. Add cash_in / cash_out to eod_reports (migration + model + backend query)
Add columns to the migration, the Rust model, and populate them from SUM(cash_in) / SUM(cash_out) on the shifts table for that day. The PDF and frontend reconciliation section will then work correctly.
2. Fix multi-shift aggregation in generate_eod_report
Replace the single-shift LIMIT 1 query with an aggregating query: SUM(opening_float) ... WHERE store_id=$1 AND opened_at::date=$2 AND status='closed'. Also aggregate cash_in and cash_out across all shifts for the day.
3. Add Lock confirmation dialog + future-date guard on generate
A simple confirm() or shadcn AlertDialog before calling lock.mutateAsync. Backend: reject date strings where date_str > today.
4. Upgrade EOD KPI row — replace Top Seller with Margin %
KPI row should be: Gross Sales · Net Sales · Gross Profit Margin % · Transactions. Move Top Seller spotlight to the Products tab only (it already exists there). Add Items Sold as a 5th card or sub-label on the Transactions card.
5. Add transactions export + make stat cards filter-aware
A "Export CSV" button in the TransactionsPanel header that calls a backend export_transactions RPC passing the current filter state. Stat cards should use the same date range / status filters as the table query.
6. Add Department filter + Margin % column to Items table
Department filter: add a department_id filter to useItems and a Select to FilterBar. Margin column: computed in the render function as ((selling - cost) / selling * 100).toFixed(1) + '%', color-coded (green > 30%, amber 15–30%, red < 15%).
